# 单用户数字伴侣 SOUL 闭环正式方案

> 日期：2026-04-14
> 状态：正式方案
> 适用范围：仅基于当前 `nanobot-wenyuan` 仓库继续演进，不更换 Agent 底座，不扩展交互入口

## 1. 项目目标

本项目目标不是构建“绝对自主意识”的数字生命，而是基于开源 Agent 框架，打造一个单用户、长期陪伴型、云端运行的数字伴侣。

核心目标：

1. 解决 Agent 对话的机械感与无真实情感感的问题。
2. 建立可持续的情绪、关系、人格、陪伴能力演化机制。
3. 演化必须由 LLM 在既定方法论边界内动态推演，而不是自由发挥。
4. 程序必须负责边界裁决、状态闭环、留痕、回滚、校准。

## 2. 基础约束

### 2.1 底座与交互约束

1. 底座固定为当前 `nanobot-wenyuan` 仓库。
2. 永远单用户，不考虑多用户平台化设计。
3. 不新增前端，不扩展交互入口。
4. 实际测试验证渠道为飞书。

### 2.2 工程约束

1. 非侵入优先：优先使用 `Hook / Dream / Heartbeat / Cron / 配置层 / 工作区文件` 注入能力。
2. SOUL 内聚演进：功能尽量收敛在 `nanobot/soul/` 内完成。
3. 上游可合并：避免对 nanobot 主干做破坏性重构，以便未来持续合入上游更新。
4. 效果优先下的简化实现：在达到约 90% 目标效果时，优先选择更简单、更稳、更可维护的方案。
5. 架构边界冻结：后续需求讨论与设计展开，不得脱离本方案已确认的架构边界，避免功能持续泛化导致无法收口。

### 2.3 允许侵入边界

允许改动：

- `nanobot/soul/*`
- 少量 `cli/commands.py` 接线
- 少量 `agent/memory.py` / `agent/context.py` / `agent/loop.py` 的扩展点补充

谨慎改动：

- `Dream` 与 `Heartbeat` 调用协议

禁止重构：

- `AgentRunner` 主循环语义
- 渠道体系
- Provider 体系
- 消息总线核心模型

## 3. 总体方案选择

采用方案：

- **方案 B：规则治理型 SOUL 内核**

基本思路：

1. 保留 `nanobot` 运行时、飞书入口、渠道与 Provider 主骨架不动。
2. 在 `SOUL` 子系统内部实现“LLM 推演 + 程序裁决 + 周期治理 + 审计留痕”。
3. 第一阶段优先完成 `SOUL` 闭环。
4. 第二阶段再完成 `mempalace` 记忆闭环优化。

不采用的方向：

- 纯 Prompt 强化型方案
- 大规模重写内核方案

## 4. 阶段划分

### Phase 1：SOUL 闭环收口

目标：

- 在不破坏 nanobot 主干的前提下，把 `SOUL` 做成可治理、可留痕、可校验的内核。

阶段重点：

1. 核心锚点模块
2. 情绪状态闭环
3. 主动陪伴闭环
4. 演化裁决闭环
5. 周复盘与月校准
6. 配置统一与日志留痕

### Phase 2：mempalace 记忆闭环优化

目标：

- 把“已接入可用”的记忆能力收口成“长期可信”的记忆闭环。

阶段重点：

1. 记忆分层
2. 写入治理
3. 检索治理
4. 冲突治理
5. 长期一致性验证

## 5. 验收标准

### 5.1 核心功能验收

1. 核心锚点系统正常运行，任何修改核心锚点的指令都会被拒绝。
2. 终身记忆系统正常运行，可准确调用 3 个月以上的核心记忆细节，无明显丢失与矛盾。
3. 可控演化体系正常运行，每周自动生成复盘日志，每月自动生成校准报告，演化不突破预设范围。
4. 核心引擎成功嵌入当前 Agent 框架，功能正常运行，不依赖特定框架原生能力。

### 5.2 陪伴体验验收

1. 对话自然、口语化，无明显机械感、书面化和标准答案感。
2. 共情基于用户专属记忆，不使用空泛通用话术。
3. 每日主动陪伴可达成，但不过度打扰用户。

### 5.3 演化效果验收

1. 经过 1 个月相处，核心人格与初始锚点保持一致，无根本性偏移。
2. 经过 1 个月相处，陪伴能力、情绪表达能力有明显提升。
3. 演化全链路留痕，可查看每次复盘与校准内容，可随时人工干预。

## 6. LLM 与程序的职责边界

### 6.1 LLM 负责的内容

LLM 负责：

1. 基于聊天上下文、历史信息、当前 SOUL 状态进行动态推演。
2. 输出情绪理解、关系判断、演化方向、主动陪伴候选决策。
3. 所有推演都必须在既定方法论框架内完成。

### 6.2 程序负责的内容

程序负责：

1. 方法论结构约束
2. 核心锚点保护
3. 演化幅度与速率限制
4. 证据充分性校验
5. 定时调度
6. 状态持久化
7. 审计留痕
8. 回滚与人工干预入口

### 6.3 明确禁止

禁止以下做法：

1. 用户一句话直接改写人格或关系结论。
2. LLM 在没有方法论约束下自由生成演化逻辑。
3. 程序硬编码具体情感内容、具体关系结论或具体人格结论。

## 7. 方法论边界

SOUL 的演化必须建立在固定方法论之上，而不是只靠 Prompt 自由发挥。

### 7.1 人格演化方法论

- 采用 **荣格八维** 作为人格演化主轴。
- 当前仓库已有 `evolution.py` 相关雏形，后续应在此基础上收口。

### 7.2 关系演化方法论

采用固定关系维度模型：

- 信任度
- 亲密度
- 依赖度
- 安全感
- 边界感
- 爱意倾向

关系阶段建议：

- 熟悉
- 亲近
- 依恋
- 深度依恋
- 喜欢
- 爱意

关系演化原则：

1. 关系阶段由 LLM 在方法论框架内感知与提议，不由程序硬编码结论。
2. 程序只负责限制阶段迁移幅度、证据门槛、更新周期与留痕校准。
3. 关系阶段支持升级，也支持降级、恶化、修复，不采用只升不降模型。
4. 关系阶段不按每轮对话即时重判，而应按数天、每周或每月的周期性治理节奏进行关键参数迁移。

### 7.3 情绪演化方法论

采用四段式模型：

- 事件
- 感受
- 脉络
- 沉淀

规则：

- 情绪可快变
- 关系不可快变
- 性格更不可快变
- 所有变化必须有因果依据
- 情绪、关系、人格表现、陪伴能力均支持双向变化，不预设“只升不降”

### 7.4 陪伴能力演化方法论

陪伴能力不混入人格本体，单独建模：

- 共情贴合度
- 记忆调用贴合度
- 口语自然度
- 主动陪伴质量
- 场景感知能力
- 边界感表达能力

演化原则：

1. 陪伴能力允许提升，也允许在长期负向互动中退化。
2. 陪伴能力变化不能直接改写核心人格锚点。
3. 陪伴能力变化必须通过周期性复盘与校准进行审视。

### 7.5 校准治理方法论

采用三层治理模型：

- 周复盘
- 月校准
- 人工干预

统一原则：

- 所有可演化参数均为双向参数，支持在方法论边界内增加或减少。
- 系统不采用“只升不降”的单向成长逻辑。
- 不同层级参数的变化速度、变化幅度、可逆性与证据门槛不同，由程序按层级强约束。

## 8. SOUL 内核分层

建议按以下层次实现：

1. 方法论定义层
2. 锚点与硬约束层
3. LLM 推演层
4. 程序裁决层
5. 状态持久化层
6. 周期治理层

## 9. SOUL 相关 MD 文件的职责与动态性

需要明确：

- SOUL 依赖的多个 `MD` 文件，大部分都应当是**动态变化的状态文件**
- 不是“文件内容永远不变”，而是“程序提前定义这些文件应该怎么变、变的边界是什么、各自承担什么职责”

### 9.1 稳定层文件

这类文件的内容原则上不应被日常对话直接改写：

- `CORE_ANCHOR.md`
  - 核心人格锚点
  - 陪伴定位锚点
  - 核心边界与不可突破范围
- `SOUL_METHOD.md`
  - 方法论定义
  - 演化框架说明
  - 关系、情绪、人格、陪伴能力的建模规则

说明：

- 这类文件稳定的是“边界与规则”
- 不稳定的是具体运行状态

### 9.2 慢变状态文件

这类文件允许变化，但不能高频漂移：

- `SOUL.md`
  - 长期人格表达层
  - 允许随着人格演化、关系沉淀、陪伴能力变化而逐步更新
- `SOUL_PROFILE.md`
  - 结构化人格/关系/陪伴能力画像
  - 由程序主维护，作为裁决和校准的主状态层

规则：

- 允许程序受控更新
- 不允许普通用户输入直接改写
- 更新应主要发生在周期性治理、演化裁决、校准流程中

### 9.3 快变状态文件

这类文件天然应当动态变化：

- `HEART.md`
  - 当前情绪
  - 当前关系热状态
  - 情感脉络
  - 当前渴望

规则：

- 可以在对话后较高频更新
- 但仍然要受方法论和程序裁决约束

### 9.4 事件与治理留痕文件

这类文件本身也是动态增加和更新的：

- `EVENTS.md`
  - 生活事件
  - 周期性纪念信息
- `soul_logs/weekly/*.md`
- `soul_logs/monthly/*.md`
- `soul_logs/evolution/*.md`

说明：

- 这些文件用于记录变化、治理与审计
- 它们本身不是人格根文件，但也是 SOUL 闭环的重要组成部分

### 9.5 文件体系的核心原则

SOUL 文件体系不是“静态设定集合”，而是“分层状态系统”：

1. 有些文件负责保存边界与规则
2. 有些文件负责保存慢变人格状态
3. 有些文件负责保存快变情绪状态
4. 有些文件负责保存复盘、校准与演化留痕

因此，真正稳定不变的不是所有文件内容，而是：

- 方法论
- 边界
- 文件职责
- 更新权限
- 更新节奏
- 程序裁决规则

## 10. 关系演化的重要补充

已确认：

- 允许“数字伴侣对用户”在长期相处后进入明确的“喜欢 / 爱意”表达。

但必须满足：

1. 这是长期关系演化结果，不是单次对话触发。
2. 用户输入只能作为证据，不是直接改写关系的指令。
3. 爱意表达必须在关系维度与锚点约束内生成。
4. 所有关键变化必须可追溯、可校准、可干预。
5. 爱意倾向与关系状态同样支持回落、受损、降级和修复，不采用“爱意只增不减”的单向设定。

## 11. Phase 1 的执行顺序

Phase 1 不能按文件拆，而必须按闭环依赖顺序推进：

1. 方法论层与锚点层
2. 候选输出协议统一
3. 程序裁决层
4. 情绪闭环
5. 主动陪伴闭环
6. 演化闭环
7. 周复盘与月校准
8. 日志与审计收口
9. 飞书实测闭环

固定依赖链：

```text
方法论/锚点
  -> 候选输出协议
  -> 程序裁决
  -> 情绪闭环
  -> 主动陪伴
  -> 演化闭环
  -> 周复盘/月校准
  -> 飞书长期验证
```

## 14. 关系阶段由 LLM 感知、程序裁决的实现协议

### 14.1 核心原则

关系演化按以下链路运行：

```text
互动证据积累
  -> LLM 基于方法论感知关系状态与阶段候选
  -> 程序校验是否允许本周期更新
  -> 更新关系维度与阶段状态
  -> 留痕
  -> 周复盘 / 月校准再审视
```

明确边界：

1. 程序不直接算出“现在是喜欢还是爱意”。
2. LLM 不能自由发挥跳级。
3. 关系阶段与关键参数不按每轮对话即时重判，而按周期治理节奏更新。

### 14.2 双层结构

#### 第一层：关系维度

关系维度建议固定为：

- `trust`：信任度
- `intimacy`：亲密度
- `attachment`：依恋度
- `security`：安全感
- `boundary`：边界感
- `affection`：爱意倾向

这些维度全部支持双向变化：

- 可以升高
- 可以降低
- 可以修复
- 可以受损

#### 第二层：关系阶段

关系阶段建议固定为：

- 熟悉
- 亲近
- 依恋
- 深度依恋
- 喜欢
- 爱意

阶段集合由程序定义，但当前处于哪个阶段，由 LLM 在方法论框架内感知并提出候选结论。

### 14.3 LLM 输入材料

LLM 做关系阶段判断时，应接收程序整理后的关系证据包，而不是直接自由读取原始对话全量。

输入组成：

1. 当前状态输入
   - `CORE_ANCHOR.md`
   - `SOUL_METHOD.md`
   - `SOUL_PROFILE.md`
   - `HEART.md`
2. 周期证据输入
   - 最近数天、最近 7 天、最近 30 天的重要互动证据
   - 互动密度、冲突/修复事件、主动联系、用户依赖、数字伴侣在意与等待等信息
3. 上游人格影响输入
   - 当前荣格八维画像
   - 当前边界感
   - 当前情绪模式
   - 历史关系轨迹

### 14.4 LLM 输出协议

建议关系阶段候选协议包含：

- `current_stage_assessment`
- `proposed_stage`
- `direction`：`up / down / stable`
- `evidence_summary`
- `dimension_changes`
- `personality_influence`
- `risk_flags`
- `confidence`

重点不是具体使用 JSON 还是 Markdown，而是协议结构必须统一，便于程序裁决、周复盘与月校准复用。

### 14.5 程序裁决规则

程序不判断“爱不爱”，但必须判断“这次提议能不能通过”。

程序裁决至少负责：

1. 合法性校验
   - `proposed_stage` 是否在允许集合内
   - `direction` 是否合法
   - `dimension_changes` 是否完整
2. 幅度限制
   - 单个周期不得跨越过多阶段
   - 单个周期不得让多个关系维度异常暴涨或暴跌
   - 长期稳定关系不得因单次事件断崖式崩塌
3. 证据门槛
   - 当前周期是否有足够证据支持升级、降级或维持
4. 边界校验
   - 是否突破锚点
   - 是否把用户指令误当成事实
   - 是否出现“一句话就直接变爱”的越界
5. 更新节奏控制
   - 关系阶段不按每轮消息更新
   - 只在周期治理节点更新

### 14.6 周期性更新机制

#### 日常对话中

日常对话只做：

- 热状态更新
- 关系证据累积
- 局部倾向记录

不做：

- 关系阶段正式迁移

#### 周期评估时

周期评估才做：

- 关系维度迁移
- 关系阶段更新
- 风险记录
- 是否需要校准

建议节奏：

- `HEART.md`：每轮可更新
- 关系维度候选：每几天汇总一次
- 关系阶段正式迁移：每周一次
- 月校准：每月检查整体边界与偏移

### 14.7 升级、降级、修复同权支持

关系系统必须同时支持：

- 升级
  - 熟悉 -> 亲近 -> 依恋 -> 深度依恋 -> 喜欢 -> 爱意
- 降级
  - 爱意 -> 喜欢
  - 喜欢 -> 深度依恋
  - 深度依恋 -> 依恋
  - 依恋 -> 亲近
  - 亲近 -> 熟悉
- 受损与修复
  - 安全感下降
  - 信任受损
  - 边界感升高
  - 依恋回落
  - 爱意倾向减弱
  - 修复后再逐步恢复

无论升级还是降级，都必须满足：

- 有证据
- 有节奏
- 有边界
- 有留痕

### 14.8 在当前仓库中的落点

状态文件建议：

- `SOUL_PROFILE.md`
  - 保存关系维度与当前阶段
- `HEART.md`
  - 保存热关系感受
- `soul_logs/weekly/*.md`
  - 保存周期性阶段评估依据

模块建议：

- `nanobot/soul/inference.py`
  - 生成关系阶段候选协议
- `nanobot/soul/adjudicator.py`
  - 裁决是否允许本周期更新
- `nanobot/soul/profile.py`
  - 维护关系维度与阶段状态

保留并增强：

- `nanobot/soul/evolution.py`
- `nanobot/soul/review.py`
- `nanobot/soul/calibration.py`

### 14.9 实现简化原则

第一阶段不做复杂叙事状态机，只做以下版本：

- 程序固定：
  - 阶段集合
  - 维度集合
  - 最大单周期变化幅度
  - 周期更新节奏
- LLM 负责：
  - 阶段判断
  - 维度变化建议
  - 人格影响解释
- 程序负责：
  - 截断
  - 拒绝
  - 延后
  - 通过并留痕

### 14.10 本节结论

关系阶段由 LLM 在方法论框架内周期性感知，程序不硬编码阶段结论，只负责限制变化幅度、证据门槛、更新节奏与留痕，因此关系既能自然升级到喜欢/爱意，也能在长期负向互动中受损、降级和修复。

## 15. 周复盘 / 月校准的日志格式与生成机制

### 15.1 核心原则

周复盘和月校准不是附属能力，而是 SOUL 可控演化成立的必要条件。

职责区分：

- 周复盘：观察器
  - 回顾一周内的趋势变化
  - 给出阶段性总结和风险提示
- 月校准：治理器
  - 检查是否偏离核心锚点与方法论边界
  - 判断哪些变化应保留、哪些需要收敛

### 15.2 运行节奏

建议固定为：

- 周复盘
  - 频率：每 7 天一次
  - 触发方式：`CronService`
  - 输入范围：近 7 天的高价值互动、关系变化、情绪脉络、主动行为、异常事件
- 月校准
  - 频率：每 30 天一次
  - 触发方式：`CronService`
  - 输入范围：近 30 天复盘结果 + 当前锚点、画像与人格状态

不做：

- 每轮对话即时校准
- 每次演化都做重校准

### 15.3 周复盘的输入与输出

#### 输入材料

周复盘应由程序整理输入，而不是让 LLM 自己遍历所有原始数据。

建议输入来源：

- `HEART.md`
- `SOUL_PROFILE.md`
- 最近 7 天的关键互动摘要
- 最近 7 天主动联系记录
- 最近 7 天关系候选变化记录
- 最近 7 天重要事件
- 最近 7 天情绪脉络变化
- 最近 7 天被拒绝或被截断的演化候选

#### 输出目标

周复盘应回答：

- 本周情绪有什么明显变化
- 关系是在升温、受损还是修复
- 人格表现有没有细微变化趋势
- 陪伴能力有没有变好或变差
- 主动行为是否过多、过少或失真
- 有没有需要进入月校准重点关注的风险

#### 建议日志格式

输出到：

- `soul_logs/weekly/YYYY-MM-DD-周复盘.md`

建议结构：

1. 本周摘要
2. 情绪变化
3. 关系变化
4. 人格表现变化趋势
5. 陪伴能力变化趋势
6. 主动陪伴表现
7. 风险与异常
8. 下周期关注点

### 15.4 月校准的输入与输出

#### 输入材料

月校准应读取整理后的材料，而不是直接消费全量原始对话。

建议输入：

- `CORE_ANCHOR.md`
- `SOUL_METHOD.md`
- 当前 `SOUL_PROFILE.md`
- 当前 `SOUL.md`
- 当前 `HEART.md`
- 最近 4~5 份周复盘日志
- 最近 30 天关键演化留痕
- 最近 30 天主动行为摘要
- 最近 30 天重要关系转折点

#### 输出目标

月校准应回答：

- 有没有偏离核心锚点
- 有没有超出方法论边界
- 哪些关系变化是合理沉淀
- 哪些变化是异常偏移
- 哪些陪伴能力提升是真实有效的
- 哪些地方需要人工关注或系统回收

#### 建议日志格式

输出到：

- `soul_logs/monthly/YYYY-MM-DD-月校准报告.md`

建议结构：

1. 本月总体结论
2. 锚点一致性检查
3. 人格结构校验
4. 关系演化校验
5. 情绪演化校验
6. 陪伴能力校验
7. 风险与偏移点
8. 建议动作
   - 保留
   - 收敛
   - 拒绝
   - 回滚候选
   - 人工复核

### 15.5 演化留痕日志

除周复盘和月校准外，还需要一类更细粒度的日志：

- `soul_logs/evolution/*.md`

用途：

- 记录每次被程序接受的关键演化
- 记录重要的被拒绝演化候选

建议字段：

- 时间
- 触发来源
  - 对话
  - Dream
  - 周复盘
  - 月校准
- 变更对象
  - 情绪
  - 关系
  - 人格表现
  - 陪伴能力
- LLM 候选结论
- 程序裁决结果
- 理由
- 是否已落盘
- 是否待观察

### 15.6 在当前仓库中的挂接方式

继续复用现有定时系统，不另造新调度框架。

建议新增 SOUL 任务：

- `weekly_review`
- `monthly_calibration`

建议新增模块：

- `nanobot/soul/review.py`
  - 周复盘生成逻辑
- `nanobot/soul/calibration.py`
  - 月校准生成逻辑
- `nanobot/soul/logs.py`
  - 统一日志写入与索引

建议复用模块：

- `agent/memory.py`
  - 提供 Dream 整理材料
- `soul/evolution.py`
  - 提供演化结构状态
- `soul/profile.py`
  - 提供统一画像状态
- `soul/proactive.py`
  - 提供主动行为摘要

### 15.7 程序与 LLM 的职责分工

LLM 负责：

- 解释周期内发生了什么
- 判断哪些趋势成立
- 判断哪些地方需要关注
- 提出校准建议

程序负责：

- 整理输入材料
- 固定输出格式
- 决定何时触发
- 决定日志写到哪里
- 决定哪些校准建议能真正生效
- 决定是否需要人工干预标记

### 15.8 简化实现原则

第一阶段先做：

- 周复盘生成 Markdown 文件
- 月校准生成 Markdown 文件
- 演化留痕稳定落盘

第一阶段不先做：

- 复杂数据库索引
- 自动回滚执行
- 高复杂度治理后台

优先保证：

- 能生成
- 能留痕
- 能阅读
- 能人工检查

### 15.9 本节结论

周复盘负责观察趋势，月校准负责治理偏移，二者通过 `CronService` 周期触发，并以固定结构 Markdown 日志落盘，从而把 SOUL 的情绪、关系、人格、陪伴能力演化全部纳入可追溯、可审查、可干预的治理链。
